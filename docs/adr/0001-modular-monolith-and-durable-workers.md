# ADR 0001: Modular monolith with durable PostgreSQL workers

- Status: accepted
- Date: 2026-07-18

## Context

The shop combines Telegram interaction, catalog, inventory, orders, payments,
wallets, delivery, broadcasts, and imports. Correct cross-module transactions
matter more than independent deployment. The Node.js reference performs
external calls inline and has no safe recovery boundary.

## Decision

Build one Go module as a modular monolith with three commands:

- `api`: Gin routes for Telegram and payment webhooks plus health/metrics.
- `worker`: outbox delivery, expiry, broadcast, and Sheet sync loops.
- `migrate`: explicit goose migration command.

PostgreSQL is the system of record and coordination mechanism. Application
services own use cases and transaction boundaries. SQL is generated by sqlc;
`pgx/v5` supplies pools, transactions, and row locking. There is no generic
repository or dependency-injection framework.

Gin is the HTTP router selected for all API endpoints. A configured
`http.Server` still wraps the Gin engine so read/write/header/idle timeouts and
graceful shutdown remain explicit.

All external side effects occur after a database commit. Retryable effects are
represented by an outbox row committed with the business change. Workers claim
rows with `FOR UPDATE SKIP LOCKED`, use bounded exponential backoff, and make
completion idempotent.

Package dependencies point inward:

```text
cmd -> transport/worker -> app -> domain
             |              |
             +-> postgres <-+
             +-> adapters (Telegram, payment, Sheets, crypto)
```

Interfaces are defined only at consuming boundaries, normally in application
services or workers. Concrete single-purpose PostgreSQL code is not hidden
behind a generic CRUD abstraction.

## Consequences

- Order/payment/inventory/wallet changes can share one PostgreSQL transaction.
- API and worker can scale independently while deploying the same artifact.
- Process crashes after commit do not lose delivery or notification intent.
- PostgreSQL load and table/index design require care because it is both data
  store and queue.
- Long-running work must be broken into short claims; no worker holds a
  transaction open during Telegram, payment, or Google network calls.
- If future load proves one boundary needs isolation, outbox contracts provide
  a migration seam without introducing a broker prematurely.

## Rejected alternatives

- A direct JavaScript-to-Go port preserves unsafe transaction and state design.
- Microservices add distributed transactions and operational cost before the
  domain boundaries are stable.
- Redis/Kafka are unnecessary in the first phase; PostgreSQL locking and outbox
  are sufficient for the expected workload.
- Long polling is convenient locally but does not meet the production webhook
  requirement.
- GORM hides critical SQL and lock semantics; pgx plus sqlc makes them explicit.
